# Connector Verify & "Verified by Organization" Certificate Plan

**Ordering:** this plan **precedes** the bot-protection/ML-moderation plan
(`repo-bot-protection-and-ml-content-moderation-plan.md`). We cannot moderate
or market connectors that are not first proven at full `pm <connector>` parity.
Goal: prove GitHub parity end-to-end with a real dev account, then turn the
existing `pm connectors certify` suite into a **signed "verified by
organization" certificate** that becomes a merge gate for every connector.

---

## 1. GitHub `gh`-parity verification (current state + checklist)

Audit of `internal/connectors/defs/github/` (read-only, no creds):

| Surface | Count | Status |
|---|---|---|
| Read streams (`streams.json`) | 37 | present |
| Write actions (`writes.json`) | 69 | present |
| `api_surface` endpoints | 509 | classified |
| ETL CLI commands implemented | 14 | parity |
| ETL CLI commands `partial` | 6 | **gap** |
| Reverse-ETL CLI commands implemented | 29 (34 mapped) | parity |
| Direct-read implemented / `planned` | 2 / 10 | **gap (by design)** |
| `unsafe_or_disallowed` / `raw_api` | 13 | intentionally blocked (destructive/admin/secret/raw) — not gaps |

**Gaps to close before "parity = true":**

1. The 6 `partial` ETL single-record lookups (`issue view`, `pr view`,
   `release view`, `workflow view`, `run view`, `ruleset view`) — these are
   redundant with their ETL list streams (records are already stream-backed),
   so the fix is either (a) promote them to `implemented` by routing to the
   stream + a record-id filter, or (b) reclassify them as `etl` aliases of the
   list stream. Verify each actually returns a record via `pm github <cmd>`.
2. The 10 `planned` direct_reads (`search repos/issues/prs/code/commits`,
   `issue/pr status`, `ruleset check`, `status`) — these are viewer-centric or
   search APIs, **out of scope for repository-scoped parity** by design. The
   decision for the certificate: mark them `out_of_scope` (not `planned`) so
   the parity matrix has no open "planned" rows.

**Parity proof (no creds needed for shape; creds for live):**

```bash
pm connectors inspect github --json            # shape + streams + writes (no creds)
go run ./cmd/connectorgen validate internal/connectors/defs --json   # 0 findings
# Live parity (needs the §3 dev credential):
pm github issues   --stream issues   --limit 5 --json   # ETL read
pm github pulls    --stream pulls    --limit 5 --json
pm github comments --stream comments --limit 5 --json
# Reverse-ETL plan/preview/approve/execute on a SAFE write in the dev repo:
pm github reverse plan   --connection github-dev --stream create_issue --record '{"title":"parity probe"}' --json
pm github reverse preview <plan-id> --json
pm github reverse run     <plan-id> --approve $PM_GITHUB_TOKEN --json
```

Acceptance for §1: 0 `partial`/`planned` rows remain; all implemented reads +
reverse-ETL writes succeed live against the dev repo; `connectorgen validate`
is clean.

## 2. Mail infra: Stalwart on the VPS over Tailscale → `github@polymetrics.ai`

**Who runs what:** the Stalwart install runs **on the VPS** (I cannot SSH
there); the steps below are exact and idempotent. I author them; you run them
on the VPS, then configure Mailspring locally and hand me a GitHub token
(never the mailbox password).

### 2.1 DNS (polymetrics.ai zone) — set before starting Stalwart

```
@                MX     10 mail.polymetrics.ai.
mail             A      <VPS-TAILSCALE-IP>     # or public IP if you terminate SMTP publicly
_stalwart._tcp   SRV    10 10 443 mail.polymetrics.ai.   # optional autodiscover
@                TXT    "v=spf1 mx -all"
_dmarc           TXT    "v=DMARC1; p=quarantine; adkim=s; aspf=s; rua=mailto:postmaster@polymetrics.ai"
# DKIM is generated by Stalwart on first run; publish the selector it prints:
<selector>._domainkey  TXT  "v=DKIM1; k=rsa; p=<base64-from-stalwart>"
```

If SMTP must not be exposed publicly, terminate on the Tailscale IP and only
relay through a smarthost (e.g. AWS SES) — `github@polymetrics.ai` then
receives via Tailscale only.

### 2.2 Stalwart (Rust, single binary) on the VPS

```bash
# On the VPS (Ubuntu/Debian), as the polymetrics user:
curl -fsSL https://github.com/stalwartlabs/stalwart/releases/latest/download/stalwart-linux-x86_64.tar.gz \
  | sudo tar -xz -C /opt/stalwart
sudo /opt/stalwart/bin/stalwart -c /opt/stalwart/etc/config.toml --init   # first run generates DKIM + admin

# Minimum config.toml additions (store data under /opt/stalwart/data):
#   [server.listen.mail]  bind = ["[::]:25","[::]:465","[::]:587"]
#   [server.listen.imap]  bind = ["[::]:993"]
#   # Tailscale-only: bind to the tail IP, or use a funnel/TLS reverse proxy
#   [storage.data] path = "/opt/stalwart/data"
#   [certificate.default] cert = "..."; key = "..."   # or Let's Encrypt for mail.polymetrics.ai

# Create the mailbox + app password (NOT the GitHub password — a dedicated app secret):
sudo /opt/stalwart/bin/stalwart -c /opt/stalwart/etc/config.toml account create \
  --name "Polymetrics GitHub" --address github@polymetrics.ai --type individual
sudo /opt/stalwart/bin/stalwart ... credential add --address github@polymetrics.ai --name mailspring
```

Harden: TLS 1.2+, disable `AUTH=PLAIN` over cleartext, enable `fail2ban` on
SMTP/IMAP, daily DKIM/SPF/DMARC review via the `postmaster` rua reports.

### 2.3 Mailspring (local) — you do this

- Add account: IMAP `mail.polymetrics.ai:993` (SSL), SMTP `:465` (SSL),
  user `github@polymetrics.ai`, the **app secret** from §2.2 (not a password).
- This mailbox is the recovery + notifications address for the dev GitHub
  account in §3. It is **not** used by `pm` at runtime — `pm` uses a PAT, never
  mailbox creds (AGENTS.md: credentials from env/stdin, never prompt).

## 3. Dev GitHub account + contributor + credential (the live parity actor)

1. Create a **new GitHub account** with `github@polymetrics.ai` as its primary
   + recovery email (you do this in the browser; I never touch the password).
2. Add the account as a **collaborator** on `polymetrics-ai/cli` **and** on a
   dedicated **dev repo** `polymetrics-ai/<dev>-parity` (the safe sandbox for
   reverse-ETL writes — never write to `cli`).
3. Create a **fine-grained PAT** scoped to the dev repo:
   `issues: write, pull_requests: write, metadata: read, contents: read`
   (no `administration`, no `secrets`, no `workflows`). Copy the token.
4. Register it with `pm` from the environment (AGENTS.md: never prompt):

```bash
export PM_GITHUB_DEV_TOKEN="<the fine-grained PAT>"   # set in your shell, not committed
pm connections create github-dev --connector github \
  --config owner=polymetrics-ai --config repo=<dev>-parity
pm credentials add github-dev-pat --connection github-dev \
  --from-env token=PM_GITHUB_DEV_TOKEN
pm connections test github-dev --json     # proves the token works, no value printed
```

Acceptance §4: `Report.Passed == true`, zero leaks, all 14 reads return rows,

## 5. Scalable per-connector flow: create app → certify → certificate

Generalize the GitHub flow to **every** connector so each earns a
**"verified by organization" certificate** only when the full suite passes.
This is the marketing-critical artifact (every connector page shows the badge).

### 5.1 Per-connector app bootstrap (scripted, not prompt)

A `scripts/connector-bootstrap.sh <connector>` (or `pm connectors bootstrap
<name>`) that, per connector, creates the provider app/credential and registers
it in `pm` from env — never printing secrets:

```bash
# Provider-specific: create an OAuth app / API key / service account in the provider
# (GitHub: fine-grained PAT; Linear/Jira/Slack: app + token; etc.) — operator step.
export PM_<UPPER>_TOKEN="<provider token>"
pm connections create <conn>-dev --connector <conn> --config <...>
pm credentials add <conn>-dev-pat --connection <conn>-dev --from-env token=PM_<UPPER>_TOKEN
pm connections test <conn>-dev --json
```

### 5.2 The certificate gate (extends `pm connectors certify`)

The certify Runner already runs the full stage suite (preflight, fixture
conformance, manual JSON, credentials, catalog, full-refresh ×3, incremental,
resume, query, write plan/preview/create/verify/cleanup) and emits
`Report.Passed` + leaks. **Extend it to emit a signed certificate on full pass:**

```json
// <root>/.polymetrics/certificates/<connector>.cert.json  (committed; the badge source)
{
  "schemaVersion": "1.0",
  "connector": "github",
  "organization": "polymetrics-ai",
  "verifiedBy": "polymetrics-ai",
  "certifiedAt": "2026-07-08T00:00:00Z",
  "expiresAt": "2026-10-08T00:00:00Z",
  "parities": { "etl": 14, "reverse_etl": 29, "direct_read": 2 },
  "gaps": [],
  "stages": ["preflight","fixture_conformance","manual_json","credentials","catalog",
             "full_refresh_append","full_refresh_overwrite","full_refresh_overwrite_deduped",
             "incremental_append","resume","query","write_plan_preview","write_create",
             "write_verify","write_cleanup"],
  "leaks": [],
  "reportHash": "sha256:<hash of the stage matrix>",
  "signature": "<org-signing-key sig over canonicalized JSON>"
}
```

**Signing:** a long-lived org Ed25519 key (`scripts/cert-sign` holds the private
key off-repo; the public key is committed to the repo + published on the
website so the badge is verifiable by anyone). The cert is emitted **only**
when `Report.Passed && len(Leaks)==0 && gaps==[]`; otherwise the file is
deleted and the connector shows **unverified**.

**Merge gate:** a `verify`-job step in `.github/workflows/connectorgen.yml`
checks that any connector whose defs changed in the PR has a current
(non-expired), signature-valid certificate; the PR cannot merge otherwise.
This is the human-gated equivalent of "no unverified connector ships."

### 5.3 Batch + rotation

```bash
pm connectors certify --all --credentials-file creds.json --parallel 4 --json   # batch
pm connectors certify --sweep                                  # clean leaked resources
# Rotate: certs expire after 90 days; re-run certify on a schedule (pm schedule, §below)
# to refresh; a connector whose cert expired shows "verification pending".
```

### 5.4 Badge + marketing

The website connector page renders a `Verified by Polymetrics` badge iff a
valid signed cert exists for that connector (generated into
`website/data/connectors.generated.json` by the certify post-step). This is the
screenshot for the blog.

## 7. Safety (AGENTS.md compliance)

- Credentials only from `--from-env`/`--value-stdin`, never prompt; never
  printed/stored/summarized.
- Reverse-ETL only plan→preview→approval→execute; dev repo only, never `cli`.
- Fine-grained PATs, least scope, no `administration`/`secrets`/`workflows`.
- `pm connectors inspect`/`connectorgen validate` never read credentials.
- Certificates carry **no secrets** — only hashes, counts, signatures.
- Stalwart app-secret ≠ GitHub password; mailbox never used by `pm` at runtime.
