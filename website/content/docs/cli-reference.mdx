This page is the public website contract for the `pm` CLI. Command/help, checked-in `docs/cli/**`, and website parity is enforced by focused tests, docs generators, and `make verify` where applicable. `pm docs validate` separately validates generated connector docs.

## Global contract

| Surface | Contract |
|---|---|
| `pm --help` | Prints the root manual. |
| `pm help <topic>` | Prints the embedded manual for a command group. |
| `pm <command> --help` | Prints that command group's embedded manual. |
| `--json` | Emits a structured envelope for commands that support machine output. |
| `--root <path>` | Selects the project root containing `.polymetrics`. |
| `--plain` | Forces the plain non-TTY path even when stdout is a terminal. |
| `--no-input` | Disables prompting and TTY UI. |
| `--progress ndjson` | Streams sanitized progress events to stderr as NDJSON; stdout remains the final output or single JSON envelope. On failures, stderr may also include the final error diagnostic. |

<Callout type="info">
  `--json` is the agent path for operational commands. A few developer-facing commands, such as `pm help`, are intentionally plain text because they return human manuals.
</Callout>

Future TTY renderers are eligible only when stdout is a TTY and `--json`, `--plain`, `--no-input`, non-empty `PM_NO_TUI`, non-empty `CI`, and `TERM=dumb` do not force the plain path. The current `cmd/pm` entrypoint keeps the plain path until a TTY renderer is wired intentionally. Use `--progress ndjson` on long-running commands when agents need live progress without polling checkpoint files.

## Configuration

`pm` resolves typed invocation configuration once per run from a bounded allowlist. Precedence is:

1. Bound global flags (`--root`, `--json`).
2. Explicit `POLYMETRICS_*` environment variables.
3. Documented `PM_*` legacy aliases when the primary variable is absent.
4. `<project-root>/.polymetrics/config.yaml`.
5. Built-in defaults.

Missing config files are allowed. Malformed YAML and invalid UI/progress flag values are validation errors (exit `3`). The `root` key in a file does not relocate config discovery for the same invocation; use `--root`, `POLYMETRICS_ROOT`, or `PM_ROOT` before loading another project. For malformed files, `--json`, `POLYMETRICS_JSON=true`, or `PM_JSON=true` selects the same single JSON `Error` envelope on stdout used by other validation errors.

Root and `json` affect invocation bootstrap. Runtime, RLM, and schedule keys are consumed by migrated runtime, worker, agent image, schedule, and RLM non-secret call sites. Worker/RLM agent Temporal execution remains opt-in: `runtime.temporal_addr` must be explicitly set by env or config file for those paths, while runtime doctor keeps local Compose defaults.

| Config key | Default | Primary env | Legacy alias |
|---|---|---|---|
| `root` | invocation root (`.`) | `POLYMETRICS_ROOT` | `PM_ROOT` |
| `json` | `false` | `POLYMETRICS_JSON` | `PM_JSON` |
| `version` | `1` | `POLYMETRICS_VERSION` | `PM_VERSION` |
| `project` | `polymetrics-local` | `POLYMETRICS_PROJECT` | `PM_PROJECT` |
| `warehouse.connector` | `warehouse` | `POLYMETRICS_WAREHOUSE_CONNECTOR` | `PM_WAREHOUSE_CONNECTOR` |
| `warehouse.path` | `.polymetrics/warehouse` | `POLYMETRICS_WAREHOUSE_PATH` | `PM_WAREHOUSE_PATH` |
| `runtime.postgres_url` | local Compose PostgreSQL DSN | `POLYMETRICS_POSTGRES_URL` | `PM_POSTGRES_URL` |
| `runtime.dragonfly_addr` | `localhost:6379` | `POLYMETRICS_DRAGONFLY_ADDR` | `PM_DRAGONFLY_ADDR` |
| `runtime.temporal_addr` | `localhost:7233` | `POLYMETRICS_TEMPORAL_ADDR` | `PM_TEMPORAL_ADDR` |
| `rlm.image` | `ghcr.io/polymetrics/rlm-agent:latest` | `POLYMETRICS_RLM_IMAGE` | `PM_RLM_IMAGE` |
| `rlm.podman_bin` | `podman` | `POLYMETRICS_PODMAN_BIN` | `PM_PODMAN_BIN` |
| `rlm.fake_runner` | `false` | `POLYMETRICS_RLM_FAKE_RUNNER` | `PM_RLM_FAKE_RUNNER` |
| `rlm.embedded_worker` | `false` | `POLYMETRICS_RLM_EMBEDDED_WORKER` | `PM_RLM_EMBEDDED_WORKER` |
| `rlm.llm.provider` | `openrouter` | `POLYMETRICS_LLM_PROVIDER` | `PM_LLM_PROVIDER` |
| `rlm.llm.base_url` | `https://openrouter.ai/api/v1` | `POLYMETRICS_LLM_BASE_URL` | `PM_LLM_BASE_URL` |
| `rlm.llm.model` | empty | `POLYMETRICS_LLM_MODEL` | `PM_LLM_MODEL` |
| `schedule.crontab_file` | empty | `POLYMETRICS_CRONTAB_FILE` | `PM_CRONTAB_FILE` |

```yaml
version: 1
project: polymetrics-local
warehouse:
  connector: warehouse
  path: .polymetrics/warehouse
runtime:
  postgres_url: postgres://localhost:15433/polymetrics?sslmode=disable
  dragonfly_addr: localhost:6379
  temporal_addr: localhost:7233
rlm:
  image: ghcr.io/polymetrics/rlm-agent:latest
  podman_bin: podman
  fake_runner: false
  embedded_worker: false
  llm:
    provider: openrouter
    base_url: https://openrouter.ai/api/v1
    model: ""
schedule:
  crontab_file: ""
```

<Callout type="warning">
  Configuration is not a secret store. User-named credential variables passed to `--from-env`, connector certification credsfile entries, `PM_LLM_API_KEY`, and provider-specific API key variables stay environment-only. Do not put secret values in `config.yaml`, docs, issues, or chat.
</Callout>

## Connector command metadata

The GitHub connector has gh-inspired command metadata for future help rendering and safe command

## Workspace commands

### `pm init`

```bash
pm init --root ./demo --json
```

Creates `.polymetrics/` under the selected root. The workspace stores state, encrypted credential references, local warehouse files, outbox files, catalogs, runs, reverse plans, flow checkpoints, and schedules.

### `pm credentials`

```bash
pm credentials add github-local \
  --connector github \
  --config repository=OWNER/REPO \
  --from-env token=GITHUB_TOKEN \
  --json

pm credentials list --json
pm credentials inspect github-local --json
pm credentials test github-local --json
pm credentials remove github-local
```

Secrets must come from environment variables or stdin. Do not pass raw token values in chat, shell history, or docs.

### `pm connections`

```bash
pm connections create github_to_warehouse \
  --source github:github-local \
  --destination warehouse:warehouse-local \
  --stream pull_requests \
  --primary-key node_id \
  --cursor updated_at \
  --sync-mode incremental_append_deduped \
  --table github_pull_requests \
  --json

pm connections list --json
```

Connections bind credentials, a stream, table, sync mode, cursor, and primary key.

### `pm catalog`

```bash
pm catalog refresh --connection github_to_warehouse --json
pm catalog show --connection github_to_warehouse --json
```

Catalog commands discover and cache stream metadata. Catalog output includes schemas and stream names, never decrypted secret values.

## Connector discovery

```bash
pm connectors list --json
pm connectors list --all --json
pm connectors catalog --capability read --json
pm connectors catalog --stage generally_available --json
pm connectors port-plan --all --json
pm connectors port-plan postgres --json
pm connectors inspect 100ms --json
pm connectors help 100ms
```

`pm connectors list` shows bundle-backed connectors. `pm connectors list --all` and `pm connectors catalog` expose the generated connector catalog with capability metadata, streams, write actions, and schema information.

## Extract, query, and write

```bash
pm etl check --connector github --config repository=OWNER/REPO --json
pm etl catalog --connector github --config repository=OWNER/REPO --json
pm etl read --connector github --stream issues --limit 10 --config repository=OWNER/REPO --json
pm etl run --connection github_to_warehouse --stream pull_requests --batch-size 500 --json --progress ndjson
pm etl status <run-id> --json
```

Use direct `check`, `catalog`, and `read` while developing a connector or testing credentials. Use `pm etl run` for configured syncs. `--progress ndjson` writes progress events to stderr; stdout stays the final human line or one JSON envelope. On failures, stderr may also include the final error diagnostic.

```bash
pm query run --table github_pull_requests --limit 20 --json
pm query run --sql "SELECT * FROM github_pull_requests LIMIT 20" --json
pm query run --table github_pull_requests --fields id,title,state --agent-mode summary --sample 3
pm query run --table github_pull_requests --fields id,title,state --agent-mode stream
```

The default build uses the local JSONL query engine. Build with the DuckDB tag when you need richer SQL support. Agent mode adds compact summary JSON and projected NDJSON output for token-sensitive callers.

```bash
pm reverse list --json
pm reverse plan prs_to_github --source-table github_pr_candidates --destination github:github-local --action create_pull_request --map title:title --map head:head --map base:base --json
pm reverse preview <plan-id> --json
pm reverse run <plan-id> --approve <approval-token> --json
pm reverse status <run-id> --json
```

Reverse ETL is always plan, preview, approval, run. JSON plan output redacts the approval token.

## Agent and automation commands

```bash
pm agent plan --request "sync sample customers" --json
pm extract --request "show active customers" --sql "SELECT * FROM customers LIMIT 20" --json
pm skills generate --dir docs/skills --json
```

`pm agent plan` returns typed command suggestions. `pm extract` classifies natural-language requests and routes simple requests to query or RLM analysis. `pm skills generate` writes agent skill files from the command and connector metadata.

```bash
pm agent image build --json
pm agent image pull --json
pm agent image ensure --json
pm rlm run --spec spec.json --in customers --out scored_customers --mode deterministic --json
pm worker status --json
pm worker serve --json
```

RLM agent mode is optional and runtime-backed. It uses Temporal plus a Podman-managed agent image; it is not part of the dependency-free quickstart.

```bash
pm flow plan --file flow.json --json
pm flow preview --file flow.json --json
pm flow run --file flow.json --force --json --progress ndjson
pm flow status nightly_leads --flows-dir .polymetrics/flows --json
pm flow list --flows-dir .polymetrics/flows --json

pm schedule create --name nightly --cron "0 2 * * *" --flow nightly_leads --json
pm schedule list --json
pm schedule install nightly --crontab --json
pm schedule remove nightly --crontab --json
```

Flows compose ETL, query, RLM, and approval-gated action steps. RLM flow steps reuse the same analyzers as `pm rlm run` and resolve relative `spec` paths from the flow manifest file. `--progress ndjson` streams flow progress events to stderr; on failures, stderr may also include the final error diagnostic. Schedules install named flows into the local scheduler backend.

## Operational commands

```bash
pm runtime doctor --json
pm perf compare --iterations 25 --json
pm perf compare --iterations 25 --runtime --json
pm perf sync-modes --records 1000 --json
```

Runtime-backed checks require PostgreSQL, DragonflyDB, and Temporal. The dependency-free path is the default.

```bash
pm docs generate --dir docs/cli --connectors-dir docs/connectors
pm docs validate --connectors-dir docs/connectors
```

`pm docs validate` validates generated connector manuals under the connector docs directory.

Runtime help, embedded command manuals, checked-in CLI markdown, and website MDX parity are covered by focused tests, generator drift checks, website docs data generation, and `make verify` where applicable.

## Exit codes

| Code | Category | Use |
|---:|---|---|
| `0` | success | Continue. |
| `2` | usage | Fix command shape. |
| `3` | validation | Fix argument values, including invalid UI/progress flag values. |
| `4` | auth | Add or rotate credentials. |
| `5` | connector | Inspect connector-specific failure. |
| `6` | runtime | Start required runtime service. |
| `7` | policy | Obtain approval or satisfy policy. |
| `1` | internal | Treat as a bug. |
