## Goal

Deliver one focused manual for every root, parent, and leaf command, while allowing the same manuals
to be assembled into a concise command tree or a full subtree/reference when explicitly requested.
All terminal, JSON, Markdown, man-page, website, golden, and future TUI-docs-viewer outputs must come
from one compiled semantic help tree.

## Locked design decisions

1. Every available canonical Cobra node has an individual manual.
2. A parent manual contains its own content and immediate child summaries only.
3. A nested command's `--help` renders that nested command, not its parent manual.
4. Full subtree concatenation is explicit; it is not the default terminal-help behavior.
5. Cobra owns structural facts: command path, usage, aliases, local/inherited flags, groups, and
   parent/child links.
6. Typed Polymetrics manual specifications own output, JSON, security, credential, approval, exit,
   example, and related-guide content.
7. A compiled manual tree feeds every renderer and generator.
8. Help resolution is metadata-only and occurs before config, telemetry, credential, runner,
   workspace, live service, write, or sweep effects.
9. Dynamic connector manuals use a provider and do not become static Cobra commands.
10. Checked-in generated bytes are deterministic and collision-checked.

## Proposed user-facing contract

| Invocation | Expected result |
|---|---|
| `pm` or `pm --help` | root manual and grouped top-level summaries |
| `pm <parent>` | parent manual, immediate child summaries, exit 0 |
| `pm <path> --help` | focused manual for the exact node |
| `pm help <path...>` | byte-identical focused manual for the exact node |
| `pm help <path...> --tree` | concise descendant path/summary outline |
| `pm help <path...> --recursive` | explicit full subtree manuals in stable preorder |
| `pm help <path...> --json` | structured `CommandManual`/`HelpTree` envelope |
| `pm docs generate --dir <dir>` | hierarchical Markdown pages plus indexes/reference data |
| `pm docs man --dir <dir>` | deterministic Section 1 man-page tree |

The exact `--tree` and `--recursive` flag names remain a small UX decision for issue #417. The
separate node/outline/subtree semantics are required.

## Expected artifacts

### Runtime architecture

- typed manual model and section types;
- command/spec binding helper;
- manual tree compiler and validator;
- exact path/alias/guide/dynamic resolver;
- text and JSON renderers;
- explicit node/tree/subtree render modes;
- metadata-only help execution boundary.

### Generated reference

- hierarchical `docs/cli/**` pages, one per command node;
- parent index pages and an optional combined searchable reference;
- deterministic `docs/man/*.1` pages via Cobra's man generator or a thin adapter;
- generated website navigation and reference data;
- updated golden transcripts and generation manifests;
- migration/update guidance for future command authors.

### Required documentation contract per command

- name, synopsis, description;
- immediate commands for parents;
- arguments and local/inherited options derived from Cobra;
- output and JSON behavior where applicable;
- environment/files where applicable;
- examples;
- security/credential/approval notes where applicable;
- exit statuses;
- parent, child, guide, and related-command links.

### RED 1 — complete command/manual inventory

Add a test that walks every available command node and fails because the current flat topic map does
not provide a unique manual for every node.

The failure report must list missing paths, duplicated parent-topic bindings, aliases, hidden-node
exceptions, and dynamic providers. Do not accept a count-only assertion.

### GREEN 1 — typed compatibility seam

Introduce the smallest model, binder, compiler, and resolver that can represent every existing flat
manual without changing bytes yet. Build from a fresh Cobra tree per invocation/test.

Keep existing `CommandManual` JSON compatibility while adding stable canonical-path metadata only if
the contract is reviewed and tests are RED first.

### RED 2 — parent versus child behavior

Add focused tests for at least:

- root versus one parent;
- `connectors` versus `connectors certify`;
- one deep TUI-era subtree present on the eventual parent head;
- alias and hidden command behavior;
- bare parent exit 0;
- invalid action usage failure.

Require parent pages to list immediate children and child pages to exclude parent-only/sibling
sections.

### GREEN 2 — focused node renderer

Render a complete manual from the exact resolved node. Derive path, usage, aliases, flags,
inheritance, groups, and children from Cobra. Render typed Polymetrics sections from the bound spec.

Make `pm <path> --help` and `pm help <path>` converge on the same renderer. Preserve JSON and
stdout/stderr rules.

### RED 3 — effect-free help

Add no-effect tests across representative risk classes:

- credential commands;
- connector inspection and certification;
- runtime/service commands;
- query/file paths;
- reverse ETL and cleanup/sweep paths;
- TUI-capable commands.

Assert no config/service initialization beyond metadata needs, no telemetry, no credential reads,
no workspace/files, no runner, no network, and no write/sweep effects.

### GREEN 3 — pre-effect help boundary

Resolve help before command pre-run/run behavior and route every help form through the manual
system. Keep malformed and invalid non-help invocations on the normal usage/error path.

### RED 4 — explicit tree composition

Test deterministic node, tree-outline, subtree, and all-reference traversal. Cover depth, stable
sorting, grouped commands, aliases, hidden commands, guides, dynamic providers, and empty
subtrees.

### GREEN 4 — tree/subtree renderers

Add explicit outline and recursive modes. Parent summaries should use each child's `Short`/summary;
recursive mode should assemble full individual manuals without re-rendering or duplicating shared
global prose unpredictably.

### RED 5 — Markdown and man generation

Add temp-directory tests proving:

- one page per canonical available node;
- hierarchical Markdown paths;
- parent/child links;
- local versus inherited option accuracy;
- man-page Section 1 naming and `SEE ALSO` links;
- deterministic reruns;
- no path traversal, symlink escape, or filename collision;
- stale generated files are detected or removed safely within the selected output root.

### GREEN 5 — deterministic generators

Generate Markdown directly from the manual tree. Use Cobra `doc.GenManTreeFromOpts` or a thin adapter
for man pages after collision preflight and typed-section enrichment. Disable wall-clock auto tags or
provide deterministic source/release dates.

Add `pm docs man` without adding a dependency beyond the approved/pinned Cobra module.

### RED 6 — website and search parity

Require the website command navigation, focused routes, combined reference, and search data to match
the generated manual manifest exactly. Missing or extra paths fail the test.

### GREEN 6 — website/reference generation

Generate website navigation/reference data from the same manual tree. Preserve hand-written
tutorials as separate content that links to stable generated command pages. Provide expandable tree
or combined-reference presentation without making every focused page enormous.

### REFACTOR — migrate every command and remove the flat map

Migrate namespaces in stable serial groups, running focused inventory/help tests after each group.
Remove a flat topic only after all of its command nodes and guide semantics are represented. Delete
legacy interception logic only when its exact replacement has RED/GREEN evidence.

The connectors/GitHub and connectors/certify pending requests are acceptance cases, not one-off
special branches in the renderer.

## Safety and non-goals

- No credentials, Keychain access, live connector calls, runtime services, reverse ETL, or sweeps
  are required to implement or verify help.
- Help must not expose approval tokens, secret-shaped examples, raw credential values, generic
  shell, generic HTTP writes, or generic SQL writes.
- Do not change command execution semantics while migrating documentation.
- Do not turn every missing leaf operand into a successful help invocation without explicit
  contract review.
- Do not add a styling/help dependency; plain, sanitized, agent-safe output remains canonical.
- Do not merge the Phase 19 PR or parent PR without the repository's human gate.
