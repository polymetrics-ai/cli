## Executive finding

The repeated cycles are not evidence that the reviewers were useless: each review prevented a
security or lifecycle defect from integrating. They do show that the review campaign was organized
for early blocker discovery, not exhaustive same-head discovery.

At the original reviewed head `4e41c2ec`, at least six top-level finding groups were already
observable: unowned late creation, quoted/multiline/client-secret/prose redaction failures,
foreground hung-hook disposal loss, flow/spaced client-secret failures, and a cancellation timer
created before admission. Those could have been found in one broader first campaign.

Seven later finding mechanisms arose in correction code: unbounded late-cleanup hooks, flow/quote/
comment state errors, multiline value-stack ownership, token-context apostrophes, and dense-line
rescanning. A reviewer could not inspect mechanisms that did not exist yet. At the behavioral level,
however, 11 of the 13 eventual probes/invariants already failed or were represented by the same
unsafe input family at the first candidate. A complete lifecycle and redaction invariant corpus
written before the first fix would therefore have prevented most later correction heads from being
declared GREEN.

The practical target is therefore **one review campaign**, not one generalist review turn: freeze
one head, run independent specialist reviews in parallel, synthesize all findings, capture one
complete test-only RED checkpoint, then make one architectural correction.

to change the object under inspection before all independent perspectives had reviewed the same

## Root cause

The primary cause was a mismatch between scope and review topology: a large security-critical diff
was given one broad behavior test batch and one generalist exact-head reviewer while review slots
were saturated. The first review returned useful blockers, but the orchestrator treated BLOCKED as
permission to mutate immediately instead of waiting for an exhaustive, parallel, same-head finding
campaign. Correction code then created new mechanisms faster than the narrow regression set could
constrain them.

## Recommended one-campaign gate

1. **Split before implementation.** Separate lifecycle/resource ownership, authority/tool policy,
   and redaction/handoff transformation when a slice contains more than one critical state machine.
2. **Require behavior-level RED.** Create compiling throwing stubs, then prove every matrix row
   fails for its intended assertion. `ERR_MODULE_NOT_FOUND` cannot close the RED gate for a broad
   feature.
3. **Build invariant matrices first.** Lifecycle must cross phase × cancellation source × hook
   outcome × expected handle counts. Redaction must cross syntax × key form × nesting × consumer ×
   harmless-control behavior and include deterministic complexity bounds.
4. **Freeze the exact head.** Do not dispatch a correction when the first reviewer finds a blocker.
   Collect all reviewer outputs against the same base/head pair first.
5. **Reserve review capacity.** Temporarily reduce implementation concurrency and run at least three
   independent `openai-codex/gpt-5.6-sol`/`xhigh` perspectives: lifecycle/timers; security/parser/
   consumers; and authority/API/performance/test quality. The parent synthesizes and deduplicates.
6. **Forbid first-blocker stopping.** Every reviewer must continue after P0/P1 findings, account for
   every changed production function and acceptance criterion, and return a completed coverage
   ledger as well as findings.
7. **Capture one complete test-only RED.** Convert the synthesized finding set into one committed
   RED checkpoint before any production correction. All consumer boundaries and harmless controls
   must be represented.
8. **Review the correction design before code.** For lifecycle, require an explicit ownership/
   cleanup state table and handle ledger. For parsers, require lexical states, newline/delimiter
   invariants, fail-closed behavior, and linear-work accounting.
9. **Keep evidence review separate from code review.** Code specialists should inspect only owned
   production/tests first; a separate verifier checks GSD artifacts, branch/scope, typecheck, RPC,
   and CI. This avoids spending the security review context on repetitive planning diffs.
10. **Use a swarm trigger.** A security-critical candidate above roughly 500 changed production
    lines, or containing two independent state machines, automatically requires the specialist
    campaign instead of a single review turn.

## Suggested reviewer contract

```text
Review frozen range BASE..HEAD. This is an exhaustive campaign, not a first-blocker check.
Do not stop after finding P0/P1 issues and do not propose or apply fixes.

For your assigned domain:
1. enumerate every state/invariant and map it to code and a test;
2. inspect every changed production function;
3. test normal, reject, throw, hang, cancellation, boundary, and adversarial cases;
4. distinguish pre-existing, candidate-introduced, and test-only gaps;
5. report all findings with a minimal reproducer and missing regression;
6. return a completed coverage ledger, including explicit CLEAN rows.

The parent will synthesize only after every specialist has returned for the same exact HEAD.
```
