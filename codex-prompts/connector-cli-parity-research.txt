## Program objective

Design a connector CLI-surface layer that makes `pm` connector docs and future connector commands
approach the quality and usefulness of famous provider CLIs, using GitHub CLI (`gh`) as the first
pilot.

The goal is not to blindly clone each vendor CLI. The goal is to make each Polymetrics connector
usable from the command line with clear command groups, usage, flags, examples, JSON output,
ETL/reverse-ETL classification, API coverage, exclusions, and safety labels.

For GitHub, use `gh` as the feature-parity baseline:

- Root help shape similar to `gh`: tagline, usage, command groups, examples, help topics, flags.
- Command groups such as issues, pull requests, repositories, releases, workflows, runs, labels,
  rulesets, secrets/variables, search, orgs, projects, gists, codespaces, discussion, auth/config,
  and other current `gh` surfaces.
- Explicitly separate API-backed commands from local workflow commands such as clone, checkout,
  browse, alias, extension, completion, and config.

For other connectors, derive a provider-specific command surface from:

- Existing `internal/connectors/defs/<name>/` streams, writes, docs, schemas, and API surface.
- Official provider API documentation.
- Official provider CLI documentation or source when a provider CLI exists.
- OpenAPI, Discovery, or similar machine-readable specs when available.

Use primary sources only for technical claims. If the source can change, browse and cite the current
official source in the artifact.

## Definitions

Use these classifications consistently.

### Command availability

- `implemented`: command maps directly to an existing stream or write action.
- `partial`: a related stream/write exists, but flags, output, scope, workflow, or behavior differ.
- `planned`: safe and useful, but no stream/write exists yet.
- `excluded`: deliberately out of scope with a written reason.
- `unsupported_local`: vendor CLI command depends on local machine workflow, browser behavior, git
  operations, shell completion, extension installation, or local config management.
- `unsupported_api`: provider API surface exists but the connector does not model it yet.
- `unsafe_or_disallowed`: violates Polymetrics safety rules, e.g. unrestricted shell/SQL/HTTP write
  behavior or destructive admin operations that should never be exposed as connector writes.
- `unknown`: source documentation is missing, unreachable, ambiguous, or needs human review.

### Command intent

- `etl`: read-oriented stream command.
- `reverse_etl`: write-oriented action that must use plan, preview, approval, execute.
- `direct_read`: read command not naturally modeled as a stream but still API-backed.
- `direct_write`: write command not naturally modeled as reverse ETL; treat as blocked unless the
  coordinator approves a safe execution model.
- `local_workflow`: local git, browser, file, config, completion, or extension command.
- `auth`: authentication/session management command.
- `config`: local or connector config command.
- `raw_api`: authenticated API escape hatch; must be constrained and approved before implementation.
- `docs_only`: useful documentation/help surface, not executable.

## JSON-first design target

Design an optional per-connector file, likely:

`internal/connectors/defs/<name>/cli_surface.json`

This file should not duplicate endpoint implementation. It should point to existing streams and
write actions where possible.

Draft shape:

```json
{
  "tagline": "Work with GitHub repositories from the command line.",
  "usage": "pm github <command> <subcommand> [flags]",
  "source_cli": {
    "name": "gh",
    "docs": "https://cli.github.com/manual/gh",
    "reference": "https://cli.github.com/manual/gh_help_reference"
  },
  "groups": [
    {
      "id": "core",
      "title": "Core Commands",
      "commands": ["issue", "pr", "repo", "release"]
    }
  ],
  "global_flags": [
    {
      "name": "json",
      "type": "boolean",
      "summary": "Write machine-readable JSON output."
    }
  ],
  "commands": [
    {
      "path": "issue list",
      "summary": "List issues",
      "intent": "etl",
      "availability": "implemented",
      "stream": "issues",
      "source_cli_path": "gh issue list",
      "flags": [
        {
          "name": "state",
          "type": "enum",
          "values": ["open", "closed", "all"],
          "maps_to": "query.state"
        }
      ],
      "examples": [
        "pm github issue list --state open --json"
      ]
    },
    {
      "path": "issue create",
      "summary": "Create an issue",
      "intent": "reverse_etl",
      "availability": "implemented",
      "write": "create_issue",
      "source_cli_path": "gh issue create",
      "risk": "creates a visible issue in the configured repository",
      "approval": "reverse ETL writes require plan, preview, approval, execute"
    }
  ],
  "help_topics": [
    {
      "name": "authentication",
      "summary": "Credential setup and supported auth modes."
    }
  ]
}
```

Also consider whether some fields should live in existing files instead:

- `metadata.json`: tagline, usage, command group summaries.
- `streams.json`: stream-level command aliases.
- `writes.json`: write-action command aliases.
- `api_surface.json`: `covered_by.command` references.

Default recommendation: keep a separate `cli_surface.json` so the command surface can evolve without
bloating core stream/write files. If runtime offline help needs the data, propose a compact generated
index instead of embedding large research ledgers.

## Parity metrics

Compute these metrics for GitHub and then every connector.

1. API endpoint coverage
   - `covered endpoints / tracked endpoints`
   - grouped by method and exclusion category.

2. CLI command coverage
   - `implemented commands / provider CLI commands`
   - include partial and unsupported categories separately.

3. Documentation coverage
   - command has usage
   - command has summary
   - command has flags
   - command has example
   - command has output notes
   - command has auth/risk notes when needed

4. Execution-model coverage
   - ETL-backed
   - reverse-ETL-backed
   - direct read candidate
   - direct write candidate
   - local workflow only
   - unsupported/disallowed

5. Implementation risk
   - no Go change
   - schema/docs only
   - engine loader change
   - CLI renderer change
   - dynamic command dispatch change
   - hook/native needed
   - unsafe/disallowed

## Implementation strategy to propose, not yet execute

Phase 0: Research only.
- Produce GitHub pilot artifacts.
- Produce schema proposal.
- Produce all-connector roster and batching plan.

Phase 1: Docs-only command surface.
- Add optional `cli_surface.json` schema.
- Add validation that command references resolve to streams/writes or explicit exclusions.
- Generate gh-like connector help previews.
- Update website docs to render the command tree.

Phase 2: Generic connector command executor.
- Add a safe generic path such as `pm connectors run <connector> <command path>`.
- Execute only commands mapped to streams or existing write actions.
- Reads default to JSON-friendly output.
- Writes use plan, preview, approval, execute.

Phase 3: Optional first-class aliases.
- Consider generated aliases like `pm github issue list`.
- Keep aliases thin wrappers around the same JSON-defined command surface.
- Do not hard-code one connector's command tree in Go.

Phase 4: Controlled raw API escape hatch, only if approved.
- Restrict to connector base URL and connector auth.
- Disallow arbitrary hosts.
- Strongly classify methods by risk.
- Require approval for mutations.
- Redact secrets.
- Do not create generic unrestricted HTTP write tooling.
